TPNT available points commands/structure

The present document describes the available settings for every device/point that suscribes to the TPNT module.
Each module that suscribes is a "device". The "device" commands apply to all the testpoints defined on that particular device.

1.- Device commands

monitor <yes | no | on | off | true | false>
Specifies if the device will be "scaned" or not. If "off", then all the suscribed points for the device will be skipped.

updaterate <rate>
Specified the rate at which the device will be scanned. Basically it means that every <rate> seconds every testpoint for this device will be reviewed. The updaterate can be specified on a per-point basis, but unless this is done all the points on the device will have this updaterate. See the updaterate parameter on the testpoints commands

alarms <yes | no | on | off | true | false>
Specifies if the alarms are active or not. If "off" the alarm check will be turned off for all the points in the device. When "on" the alarm will be back for every point that specifies so.

delay <secs>
Specifies a delay, in secs, between the time the device is sucribed and the first "scan" of the device testpoints



2.- Testpoints comands

The following commands are available for every point that is suscribed independently.

If the parent has fields, the standard way of addresing the fields is

<parent>.<fileld> = <value>

For example
samples.filter = average

It is also allowed to use the "short names" directly.
In the example
samples = average


monitor <yes | no | on | off, | true | false>
Specifies if the point will be monitored or not. If "off", the point will be skipped and none of the following applies
fileds
none

updaterate <value>
Specifies the rate, in seconds, at which the point will be monitored. By default it will use the device updaterate parameter (so, every time the device is scanned it will scan this point.
fields
none


samples
Describes what to do with the samples taken. By default it uses the raw value (no filter)

fields
field description values default short name sub-table
samples how many samples to take before applying the specified filter U16 1 samples -
filter  filter to apply to the samples average, median, lower, upper, none none filter (1)
algor algorithm to use to convert samples any defined ALGOR none algor -

(1) filter
value description
median, average calculates the "moving" median/average using the specified amount of samples (samples.samples)
lower, upper picks the lower/upper value of the samples

examples
samples.filter = average, samples.samples=10  
 /*in every update calculates the average of the last 10 points*/


log
Describes the how to log the point
fields
field description values default short name sub-table
log start or stops the logging for the point yes, no, on, off, true, false no log -
rate specifies the log rate, in seconds U16 device updaterate lograte -
file specified the file to log the point. The path must be absolute string none logfile -
format specifies the format for the logging name, value, samples, status, filter "name value" logformat -
mode specifies the log mode rate, alarm rate logmode (1)
type specifies the log type txt, html text logtype (2)

(1) mode
value description alternates names
rate logs at the rate defined by the log.rate parameter
alarm logs only if the points in under an alarm conditions. See the limits parameter.

(2) type
value description alternate names
txt file will be logges as a plain ascii text file text, ascii, plain
html file will be logged as html -

examples
log.log=yes, log.file=/home/logs/dhevars.log, log.mode=rate, log.rate=10, log.type=txt
/*log the point onto specified file every 10 seconds, as text. If more than 1 point specifies the same file, the logger will append them all in the same line if they were logged at the same time, separated */
/* by ",". for example: */
/*2006-01-12T10:22:33 CCDTEMP =160,, NECTTEMP = 230 ...*/ 


limits
Defines the limits for alarms. If the limits are used or not depends on the alarm settings

fields
field description values default short name sub-table
limits describes the limits range as "lower nominal high" "float float float" none range -
lower describes the lower allowed value float none lowerlimit -
nominal describes the "nominal" value float none nominal -
upper describes the upper allowed limit float none upperlimit -
alarm sets the alarms on or off yes, no, on, off, true, false off alarms -
alarmlatch sets the latch on the alarms (*) yes, no, off, tue, false off alarmlatch -
alarmtype sets the type of alarm none, async <err, warn, info>, mailto <...>, beep, exec <command> none alarmtype (1)
(*) The alarm latch allows to start under an outside range condition without checking alarms, untill the point it enters into the rails. After that, the alarm actions will be generated. This is used, for example, when cooling down a detector. The system starts warm, being obviously outside range. If alarms are "on" and latch is "on", no alarm will be generated untill the temperature goes into range for the first time. Then the alarms will be checked and the aalarm actions performed normally.

(1) alarmtype:
   there can be specified N alarm actions (types), using "//" to separate between actions. There is also an associated, optional sintax to specify more complex alarms.
The general sintax is:

<alarm action>  [_after_ <afvalue> <seconds | times>] [_every_ <evvalue> <seconds | times>] [_max_ <maxvalue>]

<seconds>: indicated seconds after the event was first measured to be outside limits
<times> : indicate how many times checked under repeated outside limits conditions
_after_ : indicates that the specified alarm action will start taking place after <value> spcified                       
_every_" indicated that the specified action will take place every <value> specified
_max_: indicate that the alarm action will be executed a maximum of <maxval> times

If <times> is specified rather than secs, then the actual frequency, in secs, will depend on the updaterate value for the point.
If <secods> is specified, the actual frequencey will be the closest upper multiple of the updaterate specified for the point.

Note that if many actios are specified, only 1 can have the above expresions (basically, there is only 1 set of counters for _after_, _every_ and _max_ per set of actions)
alarm actions description alternative names
async <err, war, inf> sends an asynchronous message of the specified type (error, warning or info) through the systems ' async channel asynchronous <...>
beep executes a beep
exec <command> executes the specified command on the specified device (module). The command should specify the module name also as "module <command>"
mailto <address>[-f file] sends an e-mail notification to the specified address. If "-f" is specified it will use that file as a template. See below documentation on email templates
On email templates.
The template file specified is a plain ascii test file that will be sent to the specified address. The file can have some special keywords that the system will replace in order to make the message more clear about the actual condition that gerated the alarm.

"<device>": the system will replace this keyword by the name of the device that generated the alarm
"<message>": the system will replace here the alarm message generated
"<next>": the system will replace here when the next message will be sent if the alarm condition persists.

Example of template:
"example_temple.txt " file
The <device> generated the following alarm message:
<message>
Next reminder will be sent in <next> seconds if the problem persist
Truly yours
ArcVIEW@P60

examples:
limits.lower=140, limits.upper=190, limits.nominal=160
/*this is the same as specifying:*/
limits.limits ="140 160 190"

limits.alarm=on, limits.alarmtype=async error
/*in this case an asyncronous error message will be sent to the connected client every time the alarm condition is verified*/.

limts.alarm=on, limits.alarmtype=beep // async error _after_ 5 times 
/*in this case the asynchromous message will be generated after 5 times of continuos error condition, but a beep will be generated every time. In other words, there will be 5 beeps before an async. message is actually sent up*/

limts.alarm=on, limits.alarmtype=async error // exec DHE set power off _after_ 10 times
/*in this case the asynchormous message will be sent every time, and after 10 times the condition is repeated continuously, the action "DHE set power off " will be issued.*/

limits.alarm=on, limits.alarmtype=async error // mailto elosupp@mynet.com -f example_template.txt _after_ 300 seconds _every_ 600 seconds _max_ 10
/*in this case an async. error will be sent every time, and after  300 seconds of continuos error condition, an email will be generated. Then, every 600 seconds the same email will be generated, up to a maximum of 10 times (10 emails, in this case)*/
/*If the alarm is due to the CCD temperature outside limits (201 K), measured through the DHE module, and the testpoint name is CCDTEMP, the async. error generated each time will be:*/
WARNING: CCDTEMP _above_ allowed range (201) (DHE_ASYNC)

/*and if the template is the one used on the above example, then the email generated every 600 seconds will be:*/

The DHE generated the following alarm message:
WARNING: CCDTEMP _above_ allowed range (201)
Next reminder will be sent in 600 seconds if the problem persists
Truly yours
ArcVIEW@P60

Real example
The next definition was taken from a TestPoint definition file of the SDSU DHE module on the robotic P60 telescope at Palomar. It describes the +36 volts SDSU power supply

[DEVINFO]
params="monitor=on, updaterate=60, delay=10"
 ...
/*when the device SDSU suscribes to the TPNT module, it sends the above specified device commands, in this case, monitor the device every 60 seconds, and start monitoring after 10 seconds the module is suscribed*/
...
[+HV]
type=ps
unit=volts
board=util
memory=X
address=8
algor="CONV+HV V(ADU)"
range="34 36 38"
params="monitor=on, limit.alarm=on, limit.alarmtype=async error,  log.file=/home/arclogs/pserr.log, log=on, log.mode=alarm"

The "params"  definitions are sent to the TPNT module for this testpoint when the testpoint is suscribed by the module. In this case:
monitor the point every 60 seconds (same as the device, since no "updaterate" was specified here), check for alarm conditions, and generate an asynchronous  mesage every time the  testpoint is measured to be out of range. Log the point on the specified file only if it is under an alarm condition (log.mode=alarm)


3.- External/Generic commands

These commands are meant to be asked by external clients. They give information on the available/suscribed testpoints

GETALL [types] [tpnt_type]
Retrieves information on the specified parameter
types: get a list of all the diferent types of testpoints available (temps, ps, setp, etc)
tpnt_type: get the complete list of all the testpoints of that type
if no argument is given, the int returns the full list of suscribed testpoints (regardless the device to which they may belong)

SERVER <param>
Commands to set/request the dedicated tcp/ip server
params:

enable: informs if server is available
start: starts the external server, using the rate, mode and port defined
shutdown: shutsdown the server
port [value]: informs the service port; if a value is given, changes the service port to the specified one, but this
will take effect only after restarting the server (shutdown/start)
async [on | off]: if argument is ommited, informs the server mode. If argument is given changes mode acording to it ("on" will select async mode, and "off" will select polling mode) (default: on)
rate [value]: if argument is ommited, informs the current rate. If argument is given sets it according to that. The
possible value for rate are:
            >0: if server is in async mode, it sets the rate (in msecs) at which the data stream will be sent
            <0: disables the async mode (equivalent to "async off")
            =0: if server is in async mode, it will send the async stream everytime new samples are available (so the rate will depend on the sample rate on the suscribed points) (default)

DDS <param>

enable: informs if the DDS server is available
start: starts the external DDS server, using the rate and mode specified in the SERVER command
shutdown: shutsdown the DDS server
stream: returns the name of the DDS stream to use to connect to the telemetry
