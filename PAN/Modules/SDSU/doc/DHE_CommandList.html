Command List for the Detector Head Electronics (DHE)

The following list describes the command and parameters available for ANY  detector controller module (DHE).
All the listed commands MUST begin with the prefix DHE or <module_name>. The system is case INSENSITIVE.
Note that commands on section 1 (Observer commands) are totally generic, and are the same for any DHE module. Most of the commands on section 2 (engineering commands) are generic too, but some of them are closelly related to the hardware, which implies that they must be specific for an specific DHE.
The ones described here are all the DHE commonly supported commands. Any DHE module may implement another commands, specific for that controller hardware. For those commands consult the specific controller module manual.

Command prefix: DHE

1.- Observer Commands

This is the list of commands that will be normally used during an observation. This are useful commands for the normal user.

1.1- Initialization

INIT [<dhe_confpath>]
description
    Initializes the controller. After the completion of this command, the system will be ready for taking images.  The initialization will perform basically the following steps:
    Initializes the geometry
    Initializes the controller hardware
    Initializes the GFITS and GUNSC modules
    executes any custom command(s) specified (can include any command described on this document),
    returns the status.
   
parameters
    <dhe_confpath>: path to the configuration file. The configuration file describes how the array and electronics should be initialized. 

1.2.- Exposure Setup

SET <param1>[=] <value1> [, <param2>[=] <value2>,  ... <paramN>[=]<valueN>]
description
Sets the parameter to the specified value.

parameters  
The sintax can be "param=value" (with or without blanks) or just "param val" (blank-separated). If there are multiples pairs, then they must be separated by comma (,)
     exposuretime <value> ['[s | ms]']    : sets the exposure time value for the next exposure. This value will be held untill a new <exposuretime> command is issued.                                                                   If no time unit is specified, it will assume milliseconds.
     write_to_disk <yes | no>             : write the generated image to disk or not
     multipleextensions <yes | no>     : says if the format of the output images should be multiple extension images or plain (single) fits images
     displayimage    <yes | no>           : says if you want to automatically display the image (after read or RTD)
     imagetitle <title>                         : sets the title for the image
     imagecomment <comment>        : sets a comment to appear on the headers
     imagenumber <number>            : sets the starting number for the sequence of images (see examples below)
     imagestoread  <number>             : sets the amount of images to be read in the sequence
     rootname <path+basename>        : sets the "rootname" for the image. This names INCLUDES the path (see the examples below)
     basename <basename>                : sets the basename only
     directory <path>                            : sets the directory only
     coadds <ncoadds>                        : sets the number of coadds. (IR only)

     sur (nsur>                                     : sets the number os sumple up the ramp frames (IR only)

return value
DONE on success,
ERROR: <error message> [error code] on failure

1.3.- Observation Commands

EXPOSE
description
Starts an exposition with the previously set parameters. An "exposition" may consists on N images, each of which will have the same exposure time. For the Infrared case, each image may have M coadds with K fowler samples each, or S sumple up the ramp frames.

return value
DONE on success,
ERROR: <error message> [error code] on failure


ABORT
description
Aborts all the ongoing activity, as well as all the remainig activity on the same exposure, returning the system to an "idle" state. This means that if it is in the middle of an exposure, it will abort the exposure, the following image read, and will skip all the remaining images if it was in a sequence. After the abort, the system will be idle.

return value
DONE on success
ERR: <error message> [error code] on failure


STOP
desciription
Stops the current sequence, but waits untill the current image (the one in progress) finishes normally. This means that if it is exposing, it will wait untill the current  exposure finishes, then it will read out and write that image, and then will skip the rest of the images if it was in a sequence. After the stop, the system will be idle

return value
DONE on success,
ERR: <error message> [error code] on failure


DISCARD
description
Discards current image, but continues with the next one. Thsi command will abort only the current image, and will continue with the next ones.

return value
DONE on success,
ERR: <error message> [error code] on failure


PAUSE
description
Pauses the current exposure, by closing the shutter (Optical only)

return value
DONE on success,
ERR: <error message> [error code] on failure (if not exposing it will return an error)


RESUME
description
Resumes a paused exposure, opening the shutter (Optical only)

return value
DONE on sucees
ERR: <error message> [error code] on failure    (if not paused it will return an error)


RESET <controller | pci>
description
Resets the specified device

parameters
controller         : restes the hardware controller. After this, a new "INIT" command should be sent
astropci            : resets the PCI card on the computer (if any)

2.2 DAC commands

If the controller supprts DACS commands, and DACS aredefined, then the commands should be:
DAC <SET | GET | -list> <dacname> [-unit] [-raw] [-params]
description
Sets or gets the value of a defined DAC. The defined DACs are read at initialization time from the specific DACs config file

parameters
    dacname :name of the dac. It must match some defined one
    -unit       : optiona switch which will cause the response to be returned with unit of the measurement (usually [volts])
    -raw       : optional parameter which will cause the value to be read or set in raw untis (DAC units).
    -params : optional parameter which will cause the system to return the params for that DAC (for example, memory location, board, gain, etc)
    -list        : optional parameter that will return the list of defined DACS (just the names)

return values
    GET will return the value according to the specified parameters, and ERROR: <error message> [error code] on error
    SET will return DONE o sucees, and  ERROR: <error message> [error code] on error

examples:
                  DHE DAC GET -list
                    -> DSUB
                    -> VRESET

  DHE DAC GET DSUB -unit
    -> 1.6 [v]


2.3 TP commands

If the controller suppports TP commands, and testpoints are defined, then the commands should be:

2.3.1 - TP access commands
This are commands requiered to get and set the values of the defined testpoints

TP GET [tpname | -list ] [-params [-all] [-full] [-tptype <type>]]
description
Gets the value of a defined TP. The defined TPs are read at initialization time from the specific TP config file. The GET commands brings the last value read from the hardware, but do not generate a read itself. See the READ command for "real" hardware reads. 

parameters
    tpname  :name of the testpoint. It must match some defined one
    -list       : lists all the defined testpoints (name only)
    -params : optional parameter which will cause the system to return the definition/status parameters:
                        -full: will list the full information on the testpoint
                        -all: will list the information on all the testpoints
                        -tptype <type>: will get the infor on the testpoints of the specified type only

return values
    the value according to the specified parameters on succes,
    ERROR <error message> [error code] on failure

examples:
                   DHE TP GET -list
                    -> name = CCDTEMP
                    -> name = NECKTEMP
                    -> name = HV+

                   DHE TP GET -params -tptype ps
                     -> name = HV+

                    DHE TP GET CCDTEMP 
                     -> 162.62


TP READ <tpname> 
description
Reads the value of the parameter from the hardware, and stores it. This stored value is the one that the command GET brings. This is the command that the TPNT module issues periodically when requested to poll, so the user (or external clients in general) can use GET safelly.

parameters
    tpname: name of the testpoint to be set

return values
value of the point on success
ERROR <error message> [error code] on failure



TP <SET | WRITE> <tpname> <value> ["[" unit "]"]

Sets the value of the specified testpoint to the specified value. The testpoint must have "w" or "rw" permissions on the TP file definition

parameters

    value: value to be set
    unit: the value is assumed to be in "user" units (defined on the TP config file) unless the "unit" option is added. This alows to set in DAC units rather than volts, for example.

return values

    ERROR <error message> [error code] on failure

examples:

2.5  MACRO <path> [arg1 ... argN]
description
The macro command allows to call a simple macro to generate a set of pre-defined actions from a single command. THis allows to build other commands based on existing ones. The macro files are plain ascii files, where each line has a command, omitting the DHE prefix. The commands are then separated by \n. The macro file must have an "mc" extension. The macros can take simple arguments also. See the parameters section.
The macros are useful for generating new command to perform actions that may need need several steps to complere. See the examples below.
- Macros can be nested: a macro can call another macro inside.There is  no limits on the amount of nestings.
- Any line starting with a # will be skipped, which means that comments are allowed after a # sign
- Macros accept arguments that can be used inside the macro. There is no limit to the amount of arguments.
parameters
<path>: the path to the macro file. If the path is absolute, then it will use the complete path; if the path only gives a name it will assume that the file is on the default macro directory defined on the configuration file.
[arg1...argN]: if arguments are specified, the system will replace them on the specified part on the macro. This allows to use parametric macros.
                        The arguments on the macro must be specified as
                        _ARG_[N], where N is the argument number, starting from 1
                        See the last example below ( setdac.mc)
return
DONE on succes.
"ERROR macro line number <number> [error message]" on error

examples:
1) The example below is a macro called from the init macro in order to set the right "operating point" for biases and offsets, in a system that controls a Hawaii2RG:

oper_point.mc
#sets biases operating point
dac set  VBIASPOWER 3.3
dac set VDDA 3.3
dac set VBIASGATE 2.45
dac set VRESET 0.6
#set all the offsets to the specified value
dac set OFF* 2.0

In order to call this macro "manually" (from a command line), it would be simply
DHE macro oper_point            /*the .mc can be omitted*/

2)  In a Hawaii2RG it is necessary to write to some registers in order to change some operation modes  (for example, from 1 to 32 channels readmode). If the detector is being controller using a SDSU controller, it is also necesarry to tell the DSP the new operation mode (for example, that the readmode will be in 32 channels); finally, the host software itself may need to know about it (for example, Geometry and DHE modules themselves to know how to process the incoming data) . In order to support multiple modes of operation, there are two possible approaches:

a) To support all the requiered modes on the DSP, and then just use "three letters" DSP command to change modes. The next example uses this apporach to change the detector reset mode to "line by line" reset:

line_reset.mc
#sends the SRE DSP command wich causes the DSP code to read assuming line by line reset (looping change)
memory manualcommand tim 0 none none none none SRE
#tells the H2RG mux registers to reset the data in line by line mode.
REG SET NormalModeReg {GLOBAL 0x0}

From the command line, this would be invoked as:
DHE macro line_reset

b) To support each mode using a diferent DSP file (".lod"), and load programatically another one when a diferent mode is requested. The next example uses that aproach in order to support multiple readmodes

read_32ch.mc
#loads the 32ch readmode dsp file
memory load tim file /home/arcconfig/DETECTOR/tim_32ch.lod
#tells ArcVIEW that we will use 32 channels
set readmode 32ch
#sets the external biases that set the 32ch readmode on the mux. This can be replaced by a REG command (set bits on an specific register)
dac set modectrl1 3.3
dac set modectrl2 3.3
#set multiple extension images
set image.extensions on
#sets idle mode
set idle on
#sets line by line reset by calling a macro
macro line_reset
#sets biases operating point by calling a macro
macro oper_point

This would be invoked as
DHE macro read_32ch        /*a similar read_1ch.mc call will set 1ch readmode*/.

This apporach allows to have much simpler DSP files (here we avoid the need of suporting multiple waveform tables)


3) When using a SDSU clock board as a bias generator, you may want to ensure that both HI and LO levels of a clock are changed for a particular "bias" to avoid a "switchig bias". In order to do this, you can generate a macro like this:

setdac.mc
#sets both hig and low clock levels to the same value
dac set _ARG_[1]_L  _ARG_[2]
dac set _ARG_[1]_H  _ARG_[2]

If the biases are, for example, DSUB_L and DSUB_H, then you would do
DHE macro setdac DSUB 2.0

Then the  arguments will be replaced, generating an effective output as
DHE dac set DSUB_H 2.0
DHE dac set DSUB_L 2.0


As a final note: all this functionality and much more can be reached by using just simple scripst on any scripting language. As the simplest case,  if using the pan wrapper provided in the PanVIEW/bin directory, all the macros given here can be transformed into csh scripst by just adding "pan" as prefix. For example,

oper_point.csh
#/bin/csh -f
pan dac set VBIASPOWER 3.3
pan dac set VDDA 3.3
pan dac set VBIASGATE 2.45
pan dac set  VRESET 0.6
pan dac set OFF* 2.0

So doing macros can be avoided by using csh or bash (the simplest); the macro functionality is provided just as a simplification, and does not pretend to replace the power of a script.


2.6- Memory Commands

MEMORY <action>  <parameter1> [... <parameterN> ]
description
This set of commands will perform the specified actions over the  memories of the SDSU DSPs. This commands does not perform any kind of verification about the security of the command being issued, so it must be used carefully or you can run the risk of overriding some vital part of the memory.

parameters
     <action>: actions are the actual operation to apply. The parameters will change according to it. The possible values are:
                write <board> <memtype> <address> <value> :
                                         <board> is the name of the target board (<timing | utility | pci> )
                                         <memtype> is the memory type 
                                           <address> : the memory address or range of address to write
                                            <value> : the value to write on the specified place
                read <board> <memtype> <address> :
                                            <board> is the name of the target board
                                            <memtype> is the memory type
                                            <address> : the memory address to read
                load <board> <type> <location>:     
                                           <board> is the name of the target board
                                            <type> : the type of loading you want to perform (<file | app> )
                                             <location> : where to find the source for  loading (If <file> whas specified, then it is the file path. If <app> was specified, then it is the application number)
                get <board> : gets the configured boards
               
return values
manualcommand will return a value according to the command specified.
write and load will return DONE on success, and ERR: <error message> [error code] on failure
read will return the memory value on success, and ERR: <error message> [error code] on failure

examples
            DHE MEMORY write timing X 0 132        : wites the value 132 (decimal) to the location 0 of memory X, on timing board
            DHE MEMORY write utility Y 0xA 0x132: writes the value 0x132 (hexa) to the location 10 of memory Y, on utility board
            DHE MEMORY load sequencer file tim.lod
            DHE MEMORY load timing  app 1                    /*loads the timing board with the applicatio program located on EPROM number 1 */
